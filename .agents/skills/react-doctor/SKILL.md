# React Doctor

## Purpose

Run React Doctor as a fast, external health check for React code quality in CulturaApp. Use it to catch score regressions and surface diagnostics across React correctness, effects, accessibility, performance, dead code, and architecture.

React Doctor is advisory. Interpret its findings through this repo's rules, especially Spanish UX, shared UI controls, Supabase hook boundaries, calendar layout constraints, and financial correctness.

## When to use this skill

- Use when the user mentions `react-doctor`, React Doctor, code health, React cleanup, dead code, effects, accessibility, or score regression.
- Use after making React UI or hook changes when an extra heuristic check is useful.
- Use before PR readiness work if the touched files are mostly React.

## When not to use this skill

- Do not use it instead of `npm run lint` or `npm run build` when those are required.
- Do not use it instead of `cultura-frontend-review`, `cultura-data-finance-review`, `cultura-security-privacy-review`, or `cultura-code-review` when those narrower skills apply.
- Do not treat style warnings as automatic refactor permission across unrelated files.
- Do not run remote, credentialed, destructive, or production commands from this skill.

## Inputs to inspect

- Current task or diff.
- `package.json` scripts: `doctor:react` and `doctor:react:diff`.
- Relevant React files under `src/`.
- Existing `.gitignore`, ESLint config, and any future `react-doctor.config.json`.

## Procedure

1. For a full repo health scan, run:

   ```bash
   npm run doctor:react
   ```

2. For changed-file checks before commit or PR, run:

   ```bash
   npm run doctor:react:diff
   ```

3. If the user needs every finding, run the equivalent CLI with `--verbose`.
4. Summarize score, affected categories, highest-risk files, and whether the score regressed from any known baseline.
5. Fix only findings that are in scope for the current task or clearly high-risk.
6. After fixes, rerun the relevant React Doctor command and the normal project validation that applies.

## Output format

- Score and label.
- Commands run.
- Top findings by category, with file references when actionable.
- Fixes made or deliberately deferred.
- Follow-up validation needed, if any.

## Severity / Priority Model

- CRITICAL: Findings that imply security exposure, broken auth/data boundaries, XSS risk, or user data leakage.
- HIGH: React correctness bugs, inaccessible interactive UI, broken hook behavior, or dead code that points to a missing route/export.
- MEDIUM: Performance and state/effect issues that affect visible workflows or large lists/calendars.
- LOW: Broad style, Tailwind shorthand, naming, or architecture preferences that would create noisy unrelated churn.

## Quality Bar

- Keep the scan reproducible through npm scripts.
- Keep fixes scoped to the task and grounded in user-visible or maintainability value.
- Do not silence a rule globally unless the repo has a durable reason.
- Prefer narrow suppressions with comments only after confirming the diagnostic is a false positive.

## Common Mistakes To Avoid

- Refactoring hundreds of Tailwind classes only to improve the score.
- Treating `latest` score changes as product regressions without reading the rule changes.
- Fixing dead-code findings that are actually script, test, or agent entry points without checking imports and package scripts.
- Letting React Doctor replace accessibility/manual responsive checks for calendars and modals.

## Safety Notes

The npm scripts use `npx -y react-doctor@latest` with `--offline` to skip telemetry. The CLI still downloads package code from npm when not cached, so do not run it in environments that prohibit network-installed tooling. Pin a version before using React Doctor as a CI gate.
