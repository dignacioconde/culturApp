# CACH-0035 — Rediseño financiero del Dashboard y paid_date en cobros rápidos

## Summary

El Dashboard financiero mezclaba caja real, plan mensual, deuda global, trabajos activos y métricas contables, lo que dificultaba saber qué cobrar ahora. Además, cuando se añade un cobro rápido desde la página de un proyecto marcado como ya cobrado, el campo `paid_date` se guardaba con la `start_date` del proyecto en lugar de la fecha real de cobro.

## Context

Descubierto al revisar por qué algunos cobros marcados como pagados no aparecían en el mes esperado y al rediseñar el Dashboard como herramienta de control de cobros para profesionales culturales freelance.

## Root cause

`ProjectDetail.jsx` guardaba `paid_date` con la fecha base del proyecto:

```javascript
const projectDate = project?.start_date || ''
// ...
expected_date: projectDate,          // correcto: fecha esperada de cobro
paid_date: quickIncomeForm.is_paid ? projectDate : null,  // BUG: debería ser hoy
```

`expected_date` defaulting a `projectDate` es correcto. Pero `paid_date` debe ser la fecha real de cobro, no la fecha de inicio del proyecto. Además, el Dashboard necesitaba separar plan cobrable, caja real y trabajos accionables.

## Fix

Crear una capa pura `dashboardFinance` y rediseñar el Dashboard alrededor de dos perspectivas:

- `Caja del mes`: plan cobrable del mes seleccionado.
- `Trabajos`: agrupación por proyecto/evento con contabilidad unificada.

En los cobros rápidos marcados como pagados, usar `paymentDate(new Date())` para `paid_date`.

## Impacto en datos históricos

Los cobros ya guardados con `paid_date` incorrecto quedan en el mes de `start_date` del proyecto. Corrección de datos fuera de scope de esta issue.

## Acceptance criteria

- [x] Un cobro rápido marcado como pagado guarda `paid_date = hoy` en formato `YYYY-MM-DD`.
- [x] El Dashboard muestra el plan cobrable mensual con `A cobrar = Cobrado del plan + Pendiente + Vencido`.
- [x] Los vencidos antiguos solo se arrastran al mes real actual, no a meses futuros.
- [x] Proyecto agrega ingresos directos y de eventos hijos sin duplicar eventos con proyecto.
- [x] `Trabajos` separa por perseguir, cobro futuro, ya cobrados y otros trabajos del mes.
- [x] `expected_date` sigue defaulting a la fecha prevista sin cambiar la semántica.
- [x] Tests, lint y build pasan.

## Resultado

Released en [[../releases/RELEASE-0.1.0-beta.3|RELEASE-0.1.0-beta.3]]. Se implementó rediseño de KPIs, helpers financieros testeados, UI móvil compacta y corrección de `paid_date` en cobros rápidos de proyecto y evento.
