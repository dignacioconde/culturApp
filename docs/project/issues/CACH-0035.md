# CACH-0035 — Bug paid_date en cobros rápidos de proyecto usa start_date en vez de fecha real

## Summary

Cuando se añade un cobro rápido desde la página de un proyecto marcado como ya cobrado, el campo `paid_date` se guarda con la `start_date` del proyecto en lugar de la fecha real de cobro (hoy). El Dashboard filtra `relevantIncomes` por `paid_date`, así que esos ingresos nunca aparecen en el mes correcto y los KPIs de Cobrado/Neto muestran 0,00 €.

## Context

Descubierto al revisar por qué `kpis.grossPaid` mostraba 0,00 € en marzo/abril/mayo para el proyecto Teatro Salesianos aunque tenía cobros marcados como pagados.

## Root cause

`ProjectDetail.jsx` línea ~207-213:

```javascript
const projectDate = project?.start_date || ''
// ...
expected_date: projectDate,          // correcto: fecha esperada de cobro
paid_date: quickIncomeForm.is_paid ? projectDate : null,  // BUG: debería ser hoy
```

`expected_date` defaulting a `projectDate` es correcto. Pero `paid_date` debe ser la fecha real de cobro (hoy), no la fecha de inicio del proyecto.

## Fix

En el `handleQuickIncomeSubmit` de `ProjectDetail.jsx`, usar `paymentDate(new Date())` para `paid_date`, igual que hace `useIncomes.createIncome` cuando no se provee `paid_date`.

## Impacto en datos históricos

Los cobros ya guardados con `paid_date` incorrecto quedan en el mes de `start_date` del proyecto. Corrección de datos fuera de scope de esta issue.

## Acceptance criteria

- [ ] Un cobro rápido marcado como pagado guarda `paid_date = hoy` en formato `YYYY-MM-DD`.
- [ ] El Dashboard muestra ese cobro en el mes actual correctamente.
- [ ] `expected_date` sigue defaulting a `project.start_date` sin cambios.
