# CACH-0081 — Unificar visual Lovable tras comparacion real

## Objetivo

Corregir la diferencia visual detectada al comparar Cachés con el export Lovable `artistic-rhythm`: beta 23 aplico tokens, pero todavia quedan grises legacy, radios, segmentos, cards y pantallas mixtas que hacen que la app no se perciba unificada.

## Alcance

Incluido:

- Alinear CSS base, roles tipograficos y utilidades legacy con tokens semanticos.
- Ajustar componentes compartidos `Card` y `Button` hacia el lenguaje Lovable.
- Pulir `/work`, `/dashboard`, `/settings`, `/login`, `/register` y `/onboarding` para evitar mezcla de `gray-*`, radios antiguos y controles poco coherentes.
- Mantener validacion responsive de `/dashboard`, `/work`, `/projects`, `/events`, `/settings`, `/login`, `/register` y `/onboarding`.

Fuera de alcance:

- Importar shadcn/Radix/TanStack o componentes completos de `artistic-rhythm`.
- Implementar command palette, FAB/sheets, dark mode o calendario custom.
- Cambiar Supabase, hooks de datos, rutas funcionales o formulas financieras.
- Rehacer detalles de proyecto/evento, calendario, Data o Admin en esta slice.

## Criterios de aceptacion

- [x] Los estilos base no dependen de `gray-*` para texto principal, cards, botones legacy o type roles.
- [x] `Card`, `Button`, segmentos y cards de Work/Dashboard se perciben como parte de un mismo sistema visual.
- [x] `/settings`, `/login`, `/register` y `/onboarding` no mantienen la estetica gris SaaS anterior.
- [x] No se anaden dependencias ni se importa stack Lovable.
- [x] `lint`, `build`, `pb:guard`, `verify:pr` y revision visual responsive pasan.

## Plan tecnico

Comparar `src/styles.css` y componentes `ui-cache` de `artistic-rhythm` con la app actual. Portar solo decisiones visuales inertes: tokens semanticos, radios, superficies, botones pill, segment controls, barras laterales de cards y estados base.
