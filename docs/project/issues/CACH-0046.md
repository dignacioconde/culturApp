# CACH-0046 — [Verify] Anadir verificacion por tipo de cambio

## Objetivo

Convertir la verificacion final en una matriz determinista por tipo de cambio: UI, docs, skills, agentes, datos/finanzas y release.

## Alcance

- Crear scripts npm compuestos: `verify:ci`, `verify:pr` y `pb:guard`.
- Actualizar `verification-agent` para preferir `pb:guard` y preflight de PR local.
- Incluir checks de Product Brain, contexto, skills, agentes, lint/build/test y release/smoke cuando corresponda.
- Hacer que `pb:check --strict` falle si los indices o `DIGEST.md` estan desfasados.
- Hacer visible cuando una verificacion queda saltada y por que.

Queda fuera implementar cobertura e2e completa de producto o conectar Supabase remoto.

## Criterios de aceptacion

- [x] Cada comando nuevo tiene salida clara y falla si falta una evidencia obligatoria.
- [x] Cambios docs/Product Brain ejecutan `pb:guard`, `pb:check --strict` y `git diff --check`.
- [x] Cambios skills/agentes ejecutan validaciones de symlinks, permisos y contexto.
- [x] Cambios UI quedan cubiertos por `verify:ci`; QA visual queda fuera cuando no se toca `src/`.
- [x] `verification-agent` lista comandos, evidencias y riesgos residuales.

## Validacion

- Ejecutar `npm run context:check`.
- Ejecutar `npm run verify:ci`, `npm run verify:pr -- --base origin/main`, `npm run verify:skills`, `npm run verify:agents`.
- Ejecutar `npm run lint` y `npm run build` si se toca app o tooling compartido.
- Probar que `npm run pb:check` sigue pasando.

## Resultado

Implementado un preflight local-first: `verify:ci` replica el job `app`, `verify:pr` valida diff contra base y cierre de issue cuando se pasa `--issue`, y `pb:guard` agrupa Product Brain strict, brain, index y digest.
