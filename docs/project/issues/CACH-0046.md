# CACH-0046 — [Verify] Anadir verificacion por tipo de cambio

## Objetivo

Convertir la verificacion final en una matriz determinista por tipo de cambio: UI, docs, skills, agentes, datos/finanzas y release.

## Alcance

- Crear scripts npm compuestos: `verify:docs`, `verify:skills`, `verify:agents`, `verify:ui` y `verify:release`.
- Actualizar `verification-agent` para exigir evidencias segun el tipo de cambio.
- Incluir checks de Product Brain, contexto, skills, agentes, lint/build/test y QA visual cuando corresponda.
- Hacer visible cuando una verificacion queda saltada y por que.

Queda fuera implementar cobertura e2e completa de producto o conectar Supabase remoto.

## Criterios de aceptacion

- [ ] Cada comando nuevo tiene salida clara y falla si falta una evidencia obligatoria.
- [ ] Cambios docs/Product Brain ejecutan `pb:check` y `git diff --check`.
- [ ] Cambios skills/agentes ejecutan validaciones de symlinks, permisos y contexto.
- [ ] Cambios UI definen rutas y viewports a revisar.
- [ ] `verification-agent` no se limita a una narracion; lista comandos, evidencias y riesgos residuales.

## Validacion

- Ejecutar `npm run context:check`.
- Ejecutar `npm run verify:docs`, `npm run verify:skills`, `npm run verify:agents`.
- Ejecutar `npm run lint` y `npm run build` si se toca app o tooling compartido.
- Probar que `npm run pb:check` sigue pasando.
