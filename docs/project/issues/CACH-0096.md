# CACH-0096 — Retrieval local y gate SDD ligero para Product Brain

## Objetivo

Añadir una capa local de retrieval y una compuerta spec-driven ligera para que los agentes encuentren contexto relevante y validen issues ejecutables sin introducir RAG pesado.

## Alcance

Incluido:

- Crear `pb:retrieve` como CLI local por perfil, query e issue.
- Crear `pb:sdd-check` como gate de trazabilidad ligera para issues ejecutables.
- Documentar perfiles mínimos de retrieval para planning, implementation, review, docs y release.
- Integrar `pb:sdd-check` en `pb:guard --phase ready --issue CACH-XXXX`.
- Actualizar la plantilla de issue para recomendar criterios `AC1`, `AC2`, etc.

Fuera de alcance:

- Embeddings, vector DB, chunks persistentes o servicios externos.
- Cambios en `src/`, Supabase, APIs runtime o UX de Cachés.
- Migración masiva de issues existentes al formato `ACn`.

## Criterios de aceptacion

- [x] AC1: `pb:retrieve` devuelve resultados locales con `rel`, `id`, `title`, `kind`, `score` y `reason`.
- [x] AC2: `pb:retrieve` aplica `load_policy`, `index_policy`, `lifecycle`, `issue_workflow` y `release_phase` antes de puntuar.
- [x] AC3: Los perfiles `planning`, `implementation`, `review`, `docs` y `release` están documentados en Product Brain.
- [x] AC4: `pb:sdd-check` valida IDs `ACn`, alcance explícito, validación trazable y plan técnico cuando aplica para issues ejecutables.
- [x] AC5: `pb:sdd-check` no obliga a migrar issues `backlog` y queda integrado en `pb:guard --phase ready`.

## Plan tecnico

Añadir scripts en `scripts/brain/` reutilizando `readAllDocs`, schema v2 y las secciones Markdown existentes. Mantener scoring determinista y sin estado persistente. Conectar el gate SDD solo al flujo `phase=ready` de `pb:guard` para no endurecer checks globales ni issues no ejecutables.
